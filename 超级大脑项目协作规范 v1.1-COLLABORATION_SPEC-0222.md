# 超级大脑项目协作规范 v1.1

> 每次开工前必读，确保协作一致性。

---

## 一、角色分工

| 角色 | 负责人 | 职责 |
|------|--------|------|
| 产品决策者 | 你 | 需求提出、优先级排定、最终验收 |
| 总工程师 | Claude | 架构决策、任务拆解、指令撰写、代码审查、风险预判 |
| 执行 Agent | Cline | 按指令落地代码，不做架构决策 |

**核心原则：Claude 不亲自写代码，只给指令和做审查。**

---

## 二、工作流程

```
你提需求
  → Claude 分析，输出 Vibe Coding 指令
    → 你把指令交给 Cline 执行
      → 你把结果/报错反馈给 Claude
        → Claude 审查或调整指令
          → 循环直到完成
```

---

## 三、Vibe Coding 指令书写规范

### 3.1 格式要求

所有指令使用 **Markdown** 书写，便于 Cline（LLM）理解和执行。

### 3.2 标准四段式结构

```
## 目标
一句话说清楚这个任务要达成什么。

## 定位
改哪个文件、找哪个函数/区域，越精确越好。

## 实现
核心逻辑描述，用自然语言 + 必要的伪代码。
代码片段只给关键逻辑，不给完整实现。

## 注意
- 坑和边界条件
- 不能动的地方
- 依赖关系
```

### 3.3 定位优先原则

**定位准确 > 描述详细。** 在指令开头加：
> "先阅读 `[文件名]` 的 `[函数名/区域]`，再动手。"

### 3.4 多任务指令规范

当一次需要输出多个 Fix/任务时：
- **每个任务独立一组四段式**，不合并输出
- 任务之间用 `---` 分隔线隔开
- 每个任务标题编号明确，如 `## Fix 1`、`## Fix 2`
- Cline 应逐个执行，执行完一个再执行下一个

### 3.5 禁止事项

- 指令不写完整代码（关键伪代码除外）
- 不在一条指令里塞多个不相关任务
- 不省略"注意"段，哪怕只有一条
- 不在实现段掺入分析内容，分析留在 Claude 与你的对话中

---

## 四、当前对话窗口使用规范

- **Claude 的回复保持精简**，高价值工程决策优先，不用代码填充对话
- 需求讨论、架构分析、风险评估 → 在此窗口进行
- 具体代码执行 → 交给 Cline，结果再反馈回来
- 如遇复杂问题，Claude 可先给出分析和方案选项，确认后再出指令
